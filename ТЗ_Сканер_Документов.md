## 1. Общие сведения и назначение

**Наименование системы:** Веб-приложение «Интеллектуальный сканер документов» (DocScan).

**Назначение:** Модуль позволяет пользователю сфотографировать бумажный документ камерой мобильного устройства прямо в браузере, автоматически обнаружить границы документа, исправить перспективные искажения, оптимизировать изображение и отправить результат на сервер.

**Ключевое архитектурное решение:** вся вычислительная нагрузка (детекция, коррекция, сжатие) выполняется **на стороне клиента** в браузере. Сервер выполняет только приём и сохранение готового файла. Это позволяет развернуть приложение на стандартном виртуальном (shared) хостинге без дорогостоящей серверной инфраструктуры.

---

## 2. Цели и область применения

### 2.1 Бизнес-цель

Дать пользователям возможность оцифровывать бумажные документы (чеки, справки, заявления, договоры) непосредственно через браузер смартфона — без установки нативного приложения.

### 2.2 Целевая аудитория

- Основная: пользователи мобильных устройств (iOS / Android), использующие браузер.
- Дополнительная: пользователи десктопных браузеров с подключённой веб-камерой.

### 2.3 Инфраструктурное ограничение

Хостинг — стандартный виртуальный (shared hosting). Ограничения типичного тарифа:

| Ресурс | Ограничение |
|---|---|
| RAM | 64–512 МБ |
| CPU | 1–5% от ядра |
| Timeout скрипта | 30–60 с |
| Размер POST-запроса | 2–8 МБ |

Вывод: серверная обработка изображений (PHP GD, ImageMagick) **запрещена** — слишком высокое потребление ресурсов при работе с фото высокого разрешения.

## 3. Функциональные требования

### 3.1 Инициализация камеры

- Система запрашивает доступ к задней камере устройства (`facingMode: "environment"`).
- Минимальное разрешение видеопотока: **1280×720 пикселей**.
- При наличии нескольких объективов используется основная камера.
- При отказе пользователя или отсутствии камеры — переход в режим загрузки из галереи (см. п. 3.7).

### 3.2 Детекция документа в реальном времени

- Система анализирует кадры видеопотока с частотой **не менее 10 FPS**.
- На каждом кадре выполняется поиск четырёхугольного объекта (документа).
- При обнаружении документа поверх видеопотока отображается цветная рамка-оверлей.
- Цвет рамки отражает уверенность алгоритма:
  - **Красный** — контур найден, но параметры не удовлетворяют (большой угол, низкий контраст).
  - **Жёлтый** — документ найден, незначительные отклонения.
  - **Зелёный** — документ готов к захвату.

### 3.3 Режимы захвата

**Автоматический режим:** при стабилизации зелёного оверлея на протяжении ≥ 1 секунды система автоматически выполняет захват кадра.

**Ручной режим:** пользователь нажимает кнопку затвора в любой момент.

Переключение между режимами — через элемент управления в интерфейсе.

### 3.4 Геометрическая обработка

Сразу после захвата система выполняет:
1. Обрезку изображения по найденным четырём угловым точкам.
2. Устранение перспективных искажений (warp perspective) с приведением к прямоугольному виду.
3. Нормализацию пропорций (по умолчанию — соотношение сторон A4).

### 3.5 Предпросмотр и ручное редактирование

- После обработки пользователь видит результат — выпрямленное изображение документа.
- Пользователь может вручную скорректировать положение угловых точек (drag-and-drop на исходном кадре) и повторно запустить трансформацию.
- Доступны кнопки: **«Переснять»** (возврат к видоискателю) и **«Отправить»** (переход к загрузке).

### 3.6 Сжатие и передача на сервер

- Финальное изображение конвертируется в **JPEG** с качеством **0.8** методом `canvas.toBlob()`.
- Целевой размер файла: **≤ 450 КБ** (A4, 150 DPI эквивалент).
- Файл отправляется асинхронно на сервер через **Fetch API** в формате `multipart/form-data`.
- Во время загрузки отображается прогресс-бар.

### 3.7 Fallback: загрузка из галереи

- Если камера недоступна или пользователь отказал в разрешении — отображается кнопка «Выбрать файл».
- Поддерживаемые форматы для загрузки из галереи: JPEG, PNG, HEIC.
- После выбора файла система выполняет те же шаги п. 3.4–3.6.

## 4. Нефункциональные требования

| Параметр | Требование |
|---|---|
| Время обработки | ≤ 1.5 с от захвата до отображения результата (на Samsung A54, iPhone 12) |
| Размер JS-бандла | ≤ 250 КБ в сжатом виде (gzip) |
| Частота детекции | ≥ 10 FPS на устройствах среднего класса |
| Поддержка браузеров | Chrome 100+, Safari 15.4+, Firefox 100+ |
| Протокол | Только HTTPS (MediaDevices API недоступен на HTTP) |
| Серверная нагрузка | Сервер выполняет только файловый upload, без обработки изображений |
| Размер загружаемого файла | ≤ 2 МБ (с учётом лимитов shared хостинга) |
| Время первой загрузки (FCP) | ≤ 3 с на 4G-соединении |

## 5. Технологический стек

### 5.1 Клиентская часть

| Компонент | Технология | Обоснование |
|---|---|---|
| Фреймворк | **React 18 + TypeScript** | Компонентная модель, строгая типизация |
| Сборщик | **Vite** | Быстрая сборка, нативная поддержка ESM и WASM |
| WASM-плагин | **vite-plugin-wasm + top-level-await** | Инициализация Wasm-модуля без бойлерплейта |
| Обработка изображений | **Scanic** (Rust → Wasm) | < 100 КБ, ~10 мс/кадр, GPU-ускорение |
| Захват видео | **MediaDevices API** (нативный браузерный) | Без зависимостей |
| Рендеринг | **HTML5 Canvas** через React `ref` | Прямой доступ к пикселям, трансформации |
| Стейт-менеджмент | **Zustand** | Минимальный бойлерплейт, легковесный |
| HTTP-клиент | **Fetch API** (нативный) | Без зависимостей, поддержка прогресса через XMLHttpRequest при необходимости |
| Стили | **CSS Modules** или **Tailwind CSS** | На усмотрение команды |

### 5.2 Серверная часть

| Компонент | Технология |
|---|---|
| Язык | PHP 8+ или Node.js |
| Функция | Приём `multipart/form-data`, сохранение файла |
| Библиотеки обработки изображений | **Не используются** |
| Хостинг | Любой shared хостинг с PHP/Node и HTTPS |

### 5.3 Выбор библиотеки сканирования: обоснование

| | OpenCV.js | jscanify | **Scanic** |
|---|---|---|---|
| Размер (gzip) | ~7.6–30 МБ | ≈ OpenCV.js | **< 100 КБ** |
| Скорость обработки | ~100–200 мс | ~200 мс+ | **~10 мс** |
| Трансформация | warpPerspective (CPU) | warpPerspective (CPU) | **GPU (Turbo Warp)** |
| Сложность интеграции | Высокая | Средняя | **Низкая (Modern API)** |

**Выбор: Scanic.** Единственная библиотека, укладывающаяся в бюджет бандла ≤ 250 КБ при требуемой скорости.

## 7. Алгоритм обработки документа

### Шаг 1. Захват кадра

`requestAnimationFrame` → `ctx.drawImage(videoEl, ...)` → `ImageData` → передача в Scanic.

### Шаг 2. Предобработка

Внутри Wasm-ядра (Rust):
- Перевод в градации серого.
- Применение фильтра Гаусса (размытие, подавление шума).

### Шаг 3. Детекция краёв

- Алгоритм Кэнни (Canny Edge Detector).
- Результат: бинарное изображение с картой границ.

### Шаг 4. Поиск контура документа

- Поиск всех контуров на бинарной карте.
- Фильтрация: оставляются контуры с площадью > 10% от площади кадра.
- Аппроксимация `approxPolyDP`: упрощение контура до многоугольника.
- Если аппроксимированная фигура имеет ровно **4 вершины** — классифицируется как документ.
- Из всех кандидатов выбирается контур с наибольшей площадью.

### Шаг 5. Коррекция перспективы

- Вычисление матрицы гомографии по четырём угловым точкам.
- GPU-ускоренная трансформация (Turbo Warp): разбиение на сетку треугольников и их деформация средствами Canvas/GPU.
- Результат: прямоугольное изображение документа.

### Шаг 6. Сжатие

```ts
canvas.toBlob((blob) => upload(blob), 'image/jpeg', 0.8)
```

### Шаг 7. Загрузка на сервер

```ts
const form = new FormData()
form.append('document', blob, 'scan.jpg')
fetch('/api/upload', { method: 'POST', body: form })
```

## 8. Требования к интерфейсу (UI/UX)

### 8.1 Экран видоискателя

- Видеопоток занимает всю высоту экрана.
- Полупрозрачный оверлей с рамкой-"прицелом" по центру — подсказка для позиционирования.
- Динамическая рамка меняет цвет по мере обнаружения документа (красный → жёлтый → зелёный).
- Все кнопки управления расположены внизу экрана **в зоне досягаемости большого пальца** при удержании телефона одной рукой.

### 8.2 Элементы управления

| Элемент | Расположение | Действие |
|---|---|---|
| Кнопка затвора | По центру снизу | Захват кадра |
| Переключатель авто/ручной | Слева от затвора | Смена режима захвата |
| Вспышка (если поддерживается) | Справа от затвора | Включение torch |
| Загрузка из галереи | Крайнее слева | Открыть file picker |

### 8.3 Визуальные подсказки (Toast-сообщения)

| Условие | Сообщение |
|---|---|
| Недостаточное освещение | «Слишком темно — улучшите освещение» |
| Большой угол наклона | «Держите телефон параллельно документу» |
| Документ не найден > 5 с | «Наведите камеру на документ» |
| Дрожание | «Держите телефон неподвижно» |

### 8.4 Индикаторы процесса

- После нажатия затвора — spinner поверх превью до окончания трансформации.
- Во время загрузки файла — прогресс-бар с процентом.

### 8.5 Первый запуск (Coach Marks)

- При первом открытии отображаются 2–3 полупрозрачные подсказки:
  1. «Наведите камеру на документ»
  2. «Рамка позеленеет — документ готов»
  3. «Нажмите кнопку или подождите — мы сделаем фото автоматически»
- Подсказки закрываются нажатием на любую область. Повторно не показываются (флаг в `localStorage`).

### 8.6 Экран предпросмотра

- Показывает выпрямленное изображение документа.
- Кнопка «Переснять» — возврат к видоискателю.
- Кнопка «Отправить» — запуск загрузки.
- Ручная корректировка углов: перетаскивание угловых точек на исходном кадре (переключение режима).

## 9. Просмотрщик документов

### 9.1 Назначение

После успешной загрузки пользователь должен иметь возможность просмотреть ранее отсканированные документы прямо в приложении — без скачивания файлов.

### 9.2 Список документов

- Отображается в виде **сетки карточек** (grid) с превью-изображением.
- Каждая карточка содержит:
  - Миниатюра документа (thumbnail ~200×280px).
  - Дата и время сканирования.
  - Имя файла или пользовательское название (редактируемое).
  - Иконки действий: просмотр, скачать, удалить.
- Сортировка по умолчанию: новые сверху.
- Поиск по названию документа.
- Пагинация или бесконечная прокрутка (на выбор команды).

### 9.3 Полноэкранный просмотр

При нажатии на карточку открывается **модальное окно / lightbox** с полноэкранным просмотром:

| Элемент | Описание |
|---|---|
| Изображение | Отображается на всю доступную область с сохранением пропорций |
| Зум | Pinch-to-zoom (мобильные) и колёсико мыши (десктоп) |
| Панорамирование | Drag для перемещения при увеличенном масштабе |
| Кнопки управления | «Закрыть», «Скачать», «Удалить», «Переименовать» |
| Навигация | Стрелки «Следующий» / «Предыдущий» для листания без закрытия |
| Быстрый зум | Кнопки «+», «−», «Сбросить масштаб», «По ширине», «На весь экран» |

### 9.4 Мобильные жесты

- **Pinch-to-zoom:** увеличение и уменьшение масштаба двумя пальцами.
- **Double-tap:** переключение между 100% и fit-to-screen.
- **Swipe влево/вправо:** переход к следующему/предыдущему документу.
- **Swipe вниз:** закрытие просмотрщика.

### 9.5 Компонент `DocumentViewer`

```
src/components/Viewer/
├── DocumentList.tsx      # Сетка карточек со списком документов
├── DocumentCard.tsx      # Одна карточка с превью и действиями
├── DocumentViewer.tsx    # Модальный полноэкранный просмотрщик
├── ZoomControls.tsx      # Кнопки управления масштабом
└── ViewerToolbar.tsx     # Панель: скачать, удалить, переименовать
```

### 9.6 Технические требования к просмотрщику

- Зум реализуется через CSS `transform: scale()` + `transform-origin` для плавности.
- Touch-события обрабатываются нативно (без тяжёлых библиотек). При необходимости — **use-gesture** (< 10 КБ).
- Изображения загружаются **лениво** (`loading="lazy"` для миниатюр; полное изображение — только при открытии).
- Поддержка **клавиатурной навигации**: стрелки для смены документа, Escape для закрытия, `+`/`-` для зума.
- Максимальный зум: 400%. Минимальный: fit-to-screen.

## 10. Обработка исключительных ситуаций

| Ситуация | Механизм обработки |
|---|---|
| Белый лист на белом фоне (низкий контраст) | Рамка не зеленеет; пользователь захватывает вручную и корректирует углы на превью |
| Недостаточное освещение | Toast-сообщение + запрос на включение вспышки (torch constraint, если поддерживается браузером) |
| Слишком большой угол наклона | Toast-сообщение «Держите параллельно» |
| Ошибка инициализации WebAssembly | Автоматический откат на упрощённый JS-алгоритм OR переход в режим загрузки из галереи |
| Отказ в доступе к камере | Показ кнопки «Выбрать файл из галереи» |
| Превышение размера файла после сжатия | Повторное сжатие с quality=0.6; уведомление пользователя |
| Ошибка сети при загрузке | Toast с кнопкой «Повторить» |
| Отсутствие HTTPS | Показ страницы-заглушки с инструкцией (MediaDevices API недоступен) |

## 12. Безопасность и соответствие стандартам

### 11.1 Архитектурная безопасность

- Видеопоток и все промежуточные изображения **остаются в памяти браузера** и никогда не передаются на сервер.
- На сервер отправляется только финальный JPEG после коррекции — минимизация данных (data minimization).
- Соответствует принципам **GDPR** и **152-ФЗ** (обработка ПДн на устройстве пользователя).

### 11.2 Сетевая безопасность

- **HTTPS обязателен** — MediaDevices API классифицирован браузерами как Powerful Feature, доступный только в Secure Context.
- Рекомендуется настроить **HSTS** (HTTP Strict Transport Security) для защиты от атак типа Man-in-the-Middle.
- Рекомендуемый сертификат: **Let's Encrypt** (бесплатный, автообновляемый).

### 11.3 Серверная безопасность

- Проверка типа загружаемого файла на сервере (MIME-type + расширение).
- Ограничение максимального размера файла на уровне сервера.
- Сохранение файлов вне публично доступной директории или с уникальными именами.

## 13. Ограничения серверной части

Серверный компонент намеренно минимален. Требования:

**Разрешено:**
- Принять POST-запрос с `multipart/form-data`.
- Провалидировать MIME-тип файла.
- Сохранить файл на диск или в хранилище.
- Вернуть JSON-ответ с результатом (`{ success: true, url: "..." }`).

**Запрещено:**
- Обработка изображений (GD, ImageMagick, PIL и т.д.).
- Любые операции с пикселями изображения.
- Открытие файла в оперативную память для анализа.

**Пример минимального PHP-обработчика:**

```php
<?php
$allowed = ['image/jpeg', 'image/png'];
$file = $_FILES['document'];

if (!in_array($file['type'], $allowed)) {
    http_response_code(400);
    echo json_encode(['error' => 'Invalid file type']);
    exit;
}

$dest = __DIR__ . '/uploads/' . uniqid() . '.jpg';
move_uploaded_file($file['tmp_name'], $dest);

echo json_encode(['success' => true]);
```
